3. 뷰

실제 데이터를 저장하지 않는 가상 테이블

  1. 뷰에 대해 사용자가 질의를 요청할 떄 비로소 DBMS는 뷰 정의를 참조하여 질의를 수행하고 그 결과를 사용자에게 반환
  2. 주로 기반 테이블로부터 정의되지만 또 다른 뷰를 기반으로도 정의될 수도 있음
  • 장점
    1. 편의성: 복잡한 질의문 작성이 쉽고 간단해진다
    2. 보안성: 데이터 보안 유지가 쉽다
    3. 재사용성: 반복되는 질의문 작성에 효율적이다
    4. 독립성 스키마 변경에도 뷰 질의문은 변경할 필요가 없다

3) 활용

  1. 검색

  2. 데이터 변경
    → 특정 경우로 한정 됨

    • 변경 제한의 이유

      1. 뷰를 정의하는 ‘SELECT_검색문’의 조건이 다양하기 때문
      2. 뷰가 참조하는 기반 테이블을 변경해야 하므로 내부 처리가 어려운 경우는 변경이 허용되지 않음

      갱신이 불가능한 뷰 (입력, 수정, 삭제 제한 됨)

      1. 기반 테이블의 기본키를 포함하지 않은 뷰
      2. 집계 함수(또는 계산)결과를 초함하는 뷰
      3. DISRINCT 키워드를 적용한 결과를 포함하는 뷰
      4. GROUP BY 절을 적용한 결과를 포함하는 뷰
      5. 다수의 기반 테이블의 조인을 통해 생성된 뷰
      6. 뷰에 포함되지 않은 기반 테이블의 열이 NOT NULL인 경우

4. 인덱스

테이블 안의 데이터를 쉽고 빠르게 찾을 수 있도록 만든 데이터베이스 객체

  • 필요성
    → SQL을 실행할 때, 디스크 접근 횟수를 줄여 검색 속도를 높이기 위함
    • 대부분의 DBMS는 B-트리(Balanced tree) 구조의 인덱스를 지원
    • 장점: 루트 노드로부터 리프 노트까지의 탐색 길이가 같아서 모든 데이터에 대한 일정 수준의 검색 시간을 보장

3) 고려사항

불필요한 인덱스는 오히려 성능을 떨어뜨리고 저장 공간만 낭비할 수 있으므로 꼭 필요한 만큼만 효과적으로 생성해야 함.

  1. 인덱스 생성이 필요한 경우

    1. 기본키와 외래키의 경우, 인덱스 생성이 바람직
      → 대부분의 DBMS는 기본키에 대래서 자동으로 인덱스를 생성함
    2. WHERE 절 조건식에 자주 사용되는 테이블 열의 경우, 인덱스 생성이 바람직함
    3. 조인 조건식에 자주 사용되는 테이블 열도 인덱스 생성이 바람직함
    4. 하나의 테이블에 3 ~ 5개 정도의 인덱스가 효과적임
    5. 가변길이 문자형이나 실수형, 날짜형 열보다는 정수형, 고정길이 문자형 열에 인덱스를 생성하는 것이 바람직함
    6. ORDER BY절이나 GROUP BY절에 자주 사용되는 열의 경우, 인덱스 생성을 고려할 수 있음

  2. 인덱스 생성이 불필요한 경우

    1. 갱신이 빈번한 테이블 열의 경우, 인덱스가 바람직하지 않음
    2. 집계 함수, 내장 함수를 적용하여 열 값을 변형하는 경우, 인덱스가 효과적이지 않음
    3. ‘성별’ 같은 열처럼 도메인이 작아서 열의 선택도(selectivity)가 높을 경우, 인덱스가 바람직하지 않음
    4. 범위 검색을 하는 경우, 인덱스가 바람직하지 않음
    5. 테이블의 행 개수가 별로 없는 경우, 인덱스가 바람직하지 않음

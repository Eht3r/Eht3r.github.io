2장 OS 구조

작성일 2023-09-24 Edited on 2023-11-12 In Lecture , 2-2 , OS Disqus:
Symbols count in article: 2.3k Reading time ≈ 8 mins.

1. OS 서비스

  • 운영체제는 프로그램 및 사용자에게 프로그램과 서비스를 실행할 수 있는 환경을 제공
  • 운영체제를 바라보는 관점
    • 사용자: 사용자에게 유용한 기능 제공
    • 시스템: 시스템의 효율적 운영
  • 서비스 세트

사용자에게 유용한 기능을 제공하는 운영체제 서비스 세트

  1. 사용자 인터페이스: 대부분의 OS는 사용자 인터페이스(UI)를 포함
    → 명령줄(CLI), 그래픽 사용자 인터페이스(GUI), 터치스크린 등 다양함
  2. 프로그램 실행: 프로그램을 메모리에 로드하여 해당 프로그램을 실행하고, 정상 또는 비정상적으로 실행을 종료(오류 표시)
  3. I/O 작업(입출력 작업): 실행 중인 프로그램에는 파일 또는 I/O 장치와 관련된 I/O가 필요할 수 있음
  4. 파일 시스템 조작(파일 시스템 처리/조작): 프로그램은 파일 및 디렉터리 읽기 및 쓰기, 생성 및 삭제, 검색, 파일 목록 정보, 권한 관리가 필요
  5. 통신: 프로세스는 동일한 컴퓨터에서 또는 네트워크를 통해 컴퓨터 간에 정보를 교환할 수 있음
    → 통신은 공유 메모리 또는 메시지 전달(OS에 의해 이동되는 패킷)을 통해 이루어질 수 있음
  6. 오류 감지(오류 검출): OS는 가능한 오류를 지속적으로 인식해야 합니다.
    • CPU, 메모리 하드웨어, I/O 장치, 사용자 프로그램에서 발생할 수 있음
    • 각 오류 유형에 대해 OS는 정확하고 일관된 컴퓨팅을 보장하기 위해 적절한 조치를 취해야 함
    • 디버깅 기능(디버깅 기능)은 사용자 및 프로그래머의 효율적인 시스템 사용 능력을 크게 향상시킬 수 있음

자원 공유를 통해 시스템의 효율적인 운영을 보장하기 위한 운영체제 서비스 세트

  1. 자원 할당(자원 할당): 여러 사용자 또는 여러 작업이 동시에 실행되는 경우, 각 사용자에게 자원을 할당해야 함
    → CPU 사이클, 메인 메모리, 파일 저장소, I/O 장치 등 다양한 유형의 리소스가 있음
  2. 로깅(로깅/기록): 사용자가 얼마나 많은 컴퓨터 리소스를 사용하는지, 어떤 종류의 리소스를 사용하는지 추적
  3. 보호 및 보안(보호와 보안): 다중 사용자 또는 네트워크로 연결된 컴퓨터 시스템에 저장된 정보의 소유자는 해당 정보의 사용을 제어하기를 원할 수 있으며, 동시 프로세스가 서로 간섭하지 않아야 함
    → 보호에는 시스템 리소스에 대한 모든 액세스(접근)가 제어되도록 하는 것이 포함
    → 외부인으로부터 시스템을 보호하려면 사용자 인증이 필요하며, 잘못된 액세스 시도로부터 외부 입출력 장치를 방어하는 것까지 확장

2. 사용자와 OS 인터페이스

Command Line Interpreter(CLI): 명령어를 직접 입력할 수 있는 CLI

  • 때로는 커널에서, 때로는 시스템 프로그램에서 구현되기도 함
  • 여러 가지 버전이 구현되기도 함 + 셸
  • 주로 사용자로부터 명령을 가져와 실행
  • 때로는 명령이 내장되어 있고, 때로는 프로그램 이름만 있는 경우도 있음
    → 후자의 경우, 새로운 기능을 추가해도 셸을 수정할 필요가 없음

사용자 운영 체제 인터페이스: GUI

  • 사용자 친화적인 데스크톱 인터페이스 
    1. 보통 마우스, 키보드, 모니터 
    2. 아이콘은 파일, 프로그램, 작업 등을 나타냄
    3. 인터페이스에서 개체를 마우스 버튼으로 클릭하면 다양한 동작(정보 제공, 옵션, 기능 실행, 디렉토리(폴더라고 함) 열기 등)이 수행
    4. 제록스 파크에서 발명
  • 현재 많은 시스템에 CLI와 GUI가 모두 포함됨 
    1. Microsoft Windows는 CLI “명령” 셸이 있는 GUI
    2. Apple Mac OS X는 UNIX 커널이 아래에 있고 셸을 사용할 수 있는 “Aqua” GUI 인터페이스
    3. Unix 및 Linux에는 선택적 GUI 인터페이스(CDE, KDE, GNOME)를 갖춘 CLI

터치스크린 인터페이스

  • 터치스크린 장치에는 새로운 인터페이스가 필요
    1. 마우스를 사용할 수 없거나 원하지 않음
    2. 제스처에 기반한 작업 및 선택
    3. 텍스트 입력을 위한 가상 키보드

그 외 다른 인터페이스

→ 음성 명령, 제스처

3. System Call

OS에서 제공하는 서비스에 대한 프로그래밍 인터페이스 

  • 일반적으로 고급 언어(C 또는 C++)로 작성
  • 대부분 프로그램에서 직접 시스템 호출을 사용하기보다는 높은 수준의 API(애플리케이션 프로그래밍 인터페이스) 를 통해 액세스
  • 가장 일반적인 세 가지 API 
    1. Windows용 Win32 API
    2. POSIX 기반 시스템용 POSIX API(거의 모든 버전의 UNIX, Linux 및 Mac OS X 포함)
    3. Java 가상 머신(JVM)을 위한 Java API

     

  • 이 텍스트 전체에 사용된 시스템 호출 이름은 일반적

    한 파일의 내용을 다른 파일로 복사하기 위한 시스템 호출 시퀀스

구현

  • 일반적으로 각 시스템 호출에는 숫자가 연관
    → 시스템 호출 인터페이스는 이 번호에 따라 색인된 테이블을 유지
  • 시스템 호출 인터페이스는 OS 커널에서 의도한 시스템 호출을 호출하고 시스템 호출의 상태와 모든 반환 값을 반환
  • 호출자는 시스템 호출이 어떻게 구현되는지 알 필요가 없음
    → API를 준수하고 결과 호출로 OS가 무엇을 수행할지 이해하기만 하면 됨
    → 프로그래머가 알 수 없는 OS 인터페이스의 대부분의 세부 사항은 API에 의해 숨겨짐 런타임 지원 라이브러리(컴파일러에 포함된 라이브러리에 내장된 함수 집합)에 의해 관리됨
  • API + 시스템 호출 + OS 관계

Parameter Passing

  • 원하는 시스템 호출의 신원보다 더 많은 정보가 필요한 경우가 많음
    → 정확한 유형과 정보의 양은 OS와 호출에 따라 다릅니다. 
  • OS에 파라미터(매개변수)를 전달하는 데 사용되는 일반적인 3가지 방법
    1. 가장 간단한 방법: 레지스터에 매개변수 전달
      → 경우에 따라 레지스터보다 더 많은 파라미터가 있을 수 있음 
    2. 메모리의 블록 또는 테이블에 저장된 파라미터와 레지스터에 파라미터로 전달된 블록의 주소
      • 리눅스와 솔라리스가 취하는 접근 방식
      • 파라미터는 프로그램에 의해 스택에 배치되거나 푸시되고 운영 체제에 의해 스택에서 튀어나옵니다.

       

    3. 블록 및 스택 방식은 전달되는 파라미터의 수나 길이를 제한하지 않음

시스템 콜의 6가지 타입

1. 프로세스 제어

  • 프로세스 생성, 프로세스 종료 
  • 종료, 중단 
  • 로드, 실행
  • 프로세스 속성 가져오기, 프로세스 속성 설정 
  • 시간 대기 
  • 대기 이벤트, 신호 이벤트 
  • 메모리 할당 및 해제 
  • 오류 발생 시 메모리 덤프 
  • 버그 확인을 위한 디버거, 단일 단계 실행 
  • 프로세스 간 공유 데이터에 대한 액세스를 관리하기 위한 잠금 기능

2. 파일 관리

  • 파일 생성, 파일 삭제 
  • 파일 열기, 닫기 
  • 읽기, 쓰기, 위치 변경
  • 파일 속성 가져오기 및 설정

3. 장치 관리

  • 장치 요청, 장치 해제
  • 읽기, 쓰기, 위치 변경 
  • 장치 속성 가져오기, 장치 속성 설정
  • 논리적으로 장치 연결 또는 분리

4. 정보 유지/관리

  • 시간 또는 날짜 가져오기, 시간 또는 날짜 설정
  • 시스템 데이터 가져오기, 시스템 데이터 설정
  • 프로세스, 파일 또는 장치 속성 가져오기 및 설정하기

5. 통신

  • 통신 연결 생성, 삭제 
  • 호스트 이름 또는 프로세스 이름으로 메시지 전달 모델인 경우 클라이언트에서 서버로 메시지 보내기, 받기 
  • 공유 메모리 모델 생성 및 메모리 영역에 대한 액세스 권한 부여
  • 상태 정보 전송
  • 원격 장치 연결 및 분리

6. 보호

  • 자원 접근
  • 권한
  • 사용자 액세스 허용 및 거부

4. 시스템 서비스

시스템 프로그램 또는 시스템 유틸리티는 프로그램 개발 및 실행을 위한 편리한 환경을 제공함

  • 다음의 환경들을 제공함
    1. 파일관리
    2. 상태정보
    3. 파일변경
    4. 프로그래밍 언어 지원
    5. 프로그램 로딩과 실행
    6. 통신
    7. 백그라운드리소스
  • 운영 체제에 대한 대부분의 사용자 관점은 실제 시스템 호출이 아닌 시스템 프로그램에 의해 정의
  • 프로그램 개발 및 실행을 위한 편리한 환경 제공
  • 일부는 단순히 시스템 호출에 대한 사용자 인터페이스이지만 다른 일부는 훨씬 더 복잡
  • 종류
    1. 파일 관리: 파일 및 디렉터리 생성, 삭제, 복사, 이름 변경, 인쇄, 덤프, 목록 작성 및 일반적인 조작이 가능
    2. 상태 정보
      • 일부는 날짜, 시간, 사용 가능한 메모리 양, 디스크 공간, 사용자 수 등의 정보를 시스템에 요청
      • 기타 세부 성능, 로깅 및 디버깅 정보 제공
      • 일반적으로 이러한 프로그램은 터미널 또는 기타 출력 장치에 출력을 포맷하고 인쇄
      • 일부 시스템은 구성 정보를 저장하고 검색하는 데 사용되는 레지스트리를 구현합
    3. 파일 수정
      • 파일 생성 및 수정을 위한 텍스트 편집기
      • 파일 내용을 검색하거나 텍스트 변형을 수행하는 특수 명령어
    4. 프로그래밍 언어 지원
      • 때때로 제공되는 컴파일러, 어셈블러, 디버거 및 인터프리터
    5. 프로그램 로딩 및 실행
      • 앱솔루트 로더, 재배치 가능한 로더, 링크 편집기, 오버레이 로더, 상위 수준 및 기계어를 위한 디버깅 시스템
    6. 커뮤니케이션
      • 사용자가 서로의 화면에 메시지를 보내고, 웹 페이지를 탐색하고, 전자 메일 메시지를 보내고, 원격으로 로그인하고, 한 시스템에서 다른 시스템으로 파일을 전송할 수 있도록 허용합니다.
    7. 백그라운드 서비스
      • 부팅 시 실행
      • 일부는 시스템 시작 시 실행 후 종료
        → 일부는 시스템 부팅부터 종료 시까지 실행됨
      • 디스크 검사, 프로세스 스케줄링, 오류 로깅, 인쇄와 같은 기능 제공
        → 커널 컨텍스트가 아닌 사용자 컨텍스트에서 실행됨
      • 서비스, 서브시스템, 데몬으로 알려져 있음
    8. 시스템 프로그램과 함께 애플리케이션 프로그램이 제공됨
      → 일반적인 문제를 해결하거나 일반적인 작업을 수행함
      • 사용자가 실행
      • 일반적으로 OS의 일부로 간주되지 않음
      • 명령줄, 마우스 클릭, 손가락 찌르기 등으로 실행됨

5. Linker 및 Loaders

  • 모든 물리적 메모리 위치에 로드할 수 있도록 설계된 오브젝트 파일로 컴파일된 소스 코드
    → 재배치 가능한 오브젝트 파일
  • Linker(링커) 는 단일 바이너리 실행 파일로 결합(실행 가능한 한 단위로 결합)
    → 라이브러리도 가져옴
  • 프로그램은 바이너리 실행 파일로 보조 저장소에 존재
  • 실행을 위해 Loaders(로더) 에 의해 메모리로 가져와야 함
    → Relocation(재배치) 는 프로그램 부분에 최종 주소를 할당하고 해당 주소에 맞게 프로그램 내 코드와 데이터를 조정
  • 최신 범용 시스템은 라이브러리를 실행 파일에 링크하지 않는 대신 동적으로 링크된 라이브러리(Windows의 경우 DLLs)는 필요에 따라 로드되며, 동일한 라이브러리의 동일한 버전을 사용하는 모든 라이브러리가 공유함(한 번 로드됨)
  • 객체, 실행 파일에는 표준 형식이 있으므로 운영 체제가 해당 파일을 로드하고 시작하는 방법을 알고 있음

6. 애플리케이션이 운영 체제별로 다른 이유

  • 한 시스템에서 컴파일된 앱은 일반적으로 다른 운영 체제에서 실행할 수 없음
  • 각 운영 체제는 고유한 시스템 호출(고유한 파일 형식 등)을 제공
  • 앱은 다중 운영 체제일 수 있습니다.
    • 여러 운영 체제에서 사용할 수 있는 파이썬, 루비 같은 인터프리터 언어로 작성됨
    • 실행 중인 앱이 포함된 VM을 포함하는 언어로 작성된 앱(예: Java)
    • 표준 언어(예: C) 사용, 각 운영 체제에서 개별적으로 컴파일하여 각 운영 체제에서 실행 가능
  • 애플리케이션 바이너리 인터페이스(ABI) 는 API에 해당하는 아키텍처입니다.
    • 바이너리 코드의 다양한 구성 요소가 주어진 운영 체제에 대해 어떻게 인터페이스할 수 있는지 정의
    • 바이너리 코드의 다양한 구성요소를 정의
    • 바이너리 코드의 다양한 구성요소가 지정된 아키텍처, CPU 등에서 특정 운영체제에 대해 어떻게 인터페이스 하는지 정의
  • 각 운영체제에는 헤더, 명령어, 변수의 레이아웃을 지시하는 애플리케이션용 바이너리 형식
  • CPU는 다양한 명령어 세트를 가지고 있으며, 적절한 명령어가 포함되어 있어야 애플리케이션이 올바르게 실행될 수 있음
  • OS에 대한 System calls은 다양함
    • 특정 피연산자 및 피연산자 순서
    • 애플리케이션이 시스템 호출을 호출하는 방법
    • 시스템 호출 번호와 숫자, 그 의미 및 반환 결과
  • ARM vs X86 - 명령어 집합 아키텍처

7. 운영 체제 구조

  • 범용OS는 매우 큰 프로그램입니다.
  • 다양한 구조 방식
    • 간단한 구조: MS-DOS
    • 더 복잡한 구조: UNIX
    • 계층화: 추상화
    • 마이크로커널: Mach[^11] (/mɑːk/)

모놀리식 구조(단일 구조) - 오리지널 UNIX

  • UNIX: 하드웨어 기능의 제한으로 인해 원래의 UNIX 운영 체제는 구조가 제한적
  • UNIX OS는 분리 가능한 두 부분으로 구성
    1. 시스템 프로그램
    2. 커널
      • System Calls 인터페이스 아래 및 물리적 하드웨어 위의 모든 것으로 구성됩니다.
      • 파일 시스템, CPU 스케줄링, 메모리 관리 및 기타 OS 기능을 제공하며, 하나의 레벨에 많은 기능을 제공함

Linux 시스템 구조

  • 모놀리식
    1. 속도 및 효율성
    2. 오버헤드 없음
    3. 단순성
    4. 구현 및 확장이 어려움
  • 모놀리식 + 모듈형 설계: 런타임 중에 커널을 수정할 수 있음

계층적 접근 방식

  • 운영 체제는 여러 개의 레이어(레벨)로 나뉘며, 각 레이어는 하위 레이어 위에 구축됩니다. 최하위 레이어(레이어 0)는 하드웨어이고, 최상위 레이어(레이어 N)는 사용자 인터페이스
  • 모듈화를 통해 각 레이어는 하위 레이어(하위 계층)의 기능(작업) 및 서비스만 사용하도록 선택됨
  • 간소화된 디버깅 및 시스템 검증
    • 레이어 0은 나머지 시스템에 대한 걱정 없이 디버깅할 수 있음
    • 레이어 1에서 문제가 발견되면 해당 레이어에 오류가 있는 것
  • 여러 계층의 오버헤드로 인해 전반적인 성능이 저하됨

마이크로커널

  • 커널에서 사용자 공간으로 많은 부분을 이동
  • 마이크로커널의 머신 예시
  • 부분적으로 Mach를 기반으로 하는 Mac OS X 커널(Darwin)
  • Message passing(메시지 전달) 을 사용하여 사용자 모듈 간에 통신이 이루어짐
  • 장점
    1. 마이크로커널 확장이 더 쉬움
    2. 운영 체제를 새로운 아키텍처로 포팅하기가 더 쉬움
    3. 안정성 향상(커널 모드에서 실행되는 코드 수 감소)
    4. 보안 강화
  • 단점: 사용자 공간에서 커널 공간으로 통신할 때 성능 오버헤드 발생

모듈

  • 많은 최신 운영 체제에는 로드 가능한 커널 모듈(LKMs) 이 내재되어 있음
    • 객체 지향 접근 방식 사용
    • 각 핵심 구성 요소는 분리되어 있음
    • 각각은 알려진 인터페이스를 통해 다른 구성 요소와 통신
    • 각각은 커널 내에서 필요에 따라 로드할 수 있음
  • 핵심 서비스를 제공하고, 다른 서비스는 커널이 실행되는 동안 동적으로 구현됨
  • 전반적으로 레이어와 유사하지만 더 유연(Linux, Solaris 등)

하이브리드 시스템

  • 대부분의 최신 운영 체제는 하나의 순수한 모델이 아님
    • 하이브리드는 성능, 보안, 사용성 요구 사항을 해결하기 위해 여러 접근 방식을 결합
    • 커널 주소 공간의 Linux 및 Solaris 커널은 모놀리식이며, 기능의 동적 로딩을 위한 모듈식도 존재
    • Windows 대부분 모놀리식, 다양한 하위 시스템 성격을 위한 마이크로커널 추가
  • Apple Mac OS X 하이브리드, 계층형, Aqua UI 및 Cocoa 프로그래밍 환경
    • 아래는 Mach 마이크로커널과 BSD 유닉스 부품, I/O 키트 및 동적으로 로드 가능한 모듈(커널 익스텐션이라고 함)로 구성된 커널입니다.

안드로이드

  • 오픈 핸드셋 얼라이언스(대부분 Google)에서 개발 - 오픈 소스
  • iOS와 유사한 스택
  • 리눅스 커널을 기반으로 하지만 수정됨
    • 프로세스, 메모리, 디바이스-드라이버 관리 제공

    • 라이브러리에는 웹 브라우저(웹킷), 데이터베이스(SQLite), 멀티미디어, 소규모 라이브러리용 프레임워크가 포함됩니다.

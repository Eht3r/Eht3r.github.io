1. python에서 입력 받는 방법

파이썬에서 입력을 받는 대표적인 방법은 input() 함수를 이용하는 것이다. 모든 입력은 문자열로 간주되어 문자열이 반환된다.

이 포스팅에서 모든 출력 값에 ‘’는 문자열임을 나타내기 위함으로 실제 출력시에는 같이 출력되지 않는다.

input() 함수에 인자 값으로 문자열을 넣어주면 입력 값에 대한 설명(프롬포트 메세지)을 적을 수 있다.

1.1. 문자열 이외의 타입 입력 받기

입력 한 값은 전부 문자열로 취급하기에 정순 실수 입력을 원하면 각각 int() 와 float() 함수로 형변환을 해주어야 한다.

다만, 이 방식에는 문제가 있는데 시간이 오래 걸린다는 것이다. 백준 1247 부호 문제 처럼 입력 값이 많은 경우 시간이 오래 걸려서 시간 초과가 나올 수 있다.
어떻게 해야 이 문제를 해결 할 수 있을까? 다음 쳅터에서 확인해보자

2. input() 함수의 속도 문제 해결하기

input()함수의 단점인 속도 문제 해결을 위해서는 다른 함수를 이용해야 한다. sys.stdin.readline()을 이용하면 된다. 함수의 길이가 너무 길어서 보통 기존의 input() 함수에 이를 대입해서(대체해서) 사용한다.

이 함수는 입력을 받을때 input() 처럼 기본적으로 문자열로 인식하고 문자열을 반환하며, 한 줄을 통째로 한번에 입력받는 방식이라 속도는 빠르지만 마지막에 개행문자를 포함해서 입력을 받기에 rstrip() 함수를 사용해서 개행문자를 제거 해야 한다.

input() 함수와는 달리 프롬포트 메시지를 입력할 수 없다.

2.1. 문자열 이외의 타입 입력 받기

input() 함수에 처럼 int() 와 float()를 이용해서 각각 정수와 실수로 직접 형 변환해서 입력을 받으면 된다. 이때, 원래라면 개행문자를 신경 써야 겠지만, 형 변환시 자동으로 개행무자가 제거 되어 따로 신경 써줄 필요는 없다.

3. 정리

input() 과 sys.stdin.readline() 둘 중 뭘 써도 입력 받는데는 문제가 전혀 없으나 후자가 개행문자만 신경 쓴다면 속도가 월등히 빠르기에 앞으로는 최대한 후자를 이용하기로 한다.

2장 OS 구조

작성일 2023-09-24 Edited on 2023-11-12 In Lecture , 2-2 , OS

1. OS 서비스

  • 운영체제는 프로그램 및 사용자에게 프로그램과 서비스를 실행할 수 있는 환경을 제공
  • 운영체제를 바라보는 관점
    • 사용자: 사용자에게 유용한 기능 제공
    • 시스템: 시스템의 효율적 운영
  • 서비스 세트

사용자에게 유용한 기능을 제공하는 운영체제 서비스 세트

  1. 사용자 인터페이스: 대부분의 OS는 사용자 인터페이스(UI)를 포함

    → 명령줄(CLI), 그래픽 사용자 인터페이스(GUI), 터치스크린 등 다양함
  2. 프로그램 실행: 프로그램을 메모리에 로드하여 해당 프로그램을 실행하고, 정상 또는 비정상적으로 실행을 종료(오류 표시)
  3. I/O 작업(입출력 작업): 실행 중인 프로그램에는 파일 또는 I/O 장치와 관련된 I/O가 필요할 수 있음
  4. 파일 시스템 조작(파일 시스템 처리/조작): 프로그램은 파일 및 디렉터리 읽기 및 쓰기, 생성 및 삭제, 검색, 파일 목록 정보, 권한 관리가 필요
  5. 통신: 프로세스는 동일한 컴퓨터에서 또는 네트워크를 통해 컴퓨터 간에 정보를 교환할 수 있음

    → 통신은 공유 메모리 또는 메시지 전달(OS에 의해 이동되는 패킷)을 통해 이루어질 수 있음
  6. 오류 감지(오류 검출): OS는 가능한 오류를 지속적으로 인식해야 합니다.
    • CPU, 메모리 하드웨어, I/O 장치, 사용자 프로그램에서 발생할 수 있음
    • 각 오류 유형에 대해 OS는 정확하고 일관된 컴퓨팅을 보장하기 위해 적절한 조치를 취해야 함
    • 디버깅 기능(디버깅 기능)은 사용자 및 프로그래머의 효율적인 시스템 사용 능력을 크게 향상시킬 수 있음

자원 공유를 통해 시스템의 효율적인 운영을 보장하기 위한 운영체제 서비스 세트

  1. 자원 할당(자원 할당): 여러 사용자 또는 여러 작업이 동시에 실행되는 경우, 각 사용자에게 자원을 할당해야 함
    → CPU 사이클, 메인 메모리, 파일 저장소, I/O 장치 등 다양한 유형의 리소스가 있음
  2. 로깅(로깅/기록): 사용자가 얼마나 많은 컴퓨터 리소스를 사용하는지, 어떤 종류의 리소스를 사용하는지 추적
  3. 보호 및 보안(보호와 보안): 다중 사용자 또는 네트워크로 연결된 컴퓨터 시스템에 저장된 정보의 소유자는 해당 정보의 사용을 제어하기를 원할 수 있으며, 동시 프로세스가 서로 간섭하지 않아야 함

    → 보호에는 시스템 리소스에 대한 모든 액세스(접근)가 제어되도록 하는 것이 포함

    → 외부인으로부터 시스템을 보호하려면 사용자 인증이 필요하며, 잘못된 액세스 시도로부터 외부 입출력 장치를 방어하는 것까지 확장

2. 사용자와 OS 인터페이스

Command Line Interpreter(CLI): 명령어를 직접 입력할 수 있는 CLI

  • 때로는 커널에서, 때로는 시스템 프로그램에서 구현되기도 함
  • 여러 가지 버전이 구현되기도 함 + 셸
  • 주로 사용자로부터 명령을 가져와 실행
  • 때로는 명령이 내장되어 있고, 때로는 프로그램 이름만 있는 경우도 있음

    → 후자의 경우, 새로운 기능을 추가해도 셸을 수정할 필요가 없음

사용자 운영 체제 인터페이스: GUI

  • 사용자 친화적인 데스크톱 인터페이스 
    1. 보통 마우스, 키보드, 모니터 
    2. 아이콘은 파일, 프로그램, 작업 등을 나타냄
    3. 인터페이스에서 개체를 마우스 버튼으로 클릭하면 다양한 동작(정보 제공, 옵션, 기능 실행, 디렉토리(폴더라고 함) 열기 등)이 수행
    4. 제록스 파크에서 발명
  • 현재 많은 시스템에 CLI와 GUI가 모두 포함됨 
    1. Microsoft Windows는 CLI “명령” 셸이 있는 GUI
    2. Apple Mac OS X는 UNIX 커널이 아래에 있고 셸을 사용할 수 있는 “Aqua” GUI 인터페이스
    3. Unix 및 Linux에는 선택적 GUI 인터페이스(CDE, KDE, GNOME)를 갖춘 CLI

터치스크린 인터페이스

  • 터치스크린 장치에는 새로운 인터페이스가 필요
    1. 마우스를 사용할 수 없거나 원하지 않음
    2. 제스처에 기반한 작업 및 선택
    3. 텍스트 입력을 위한 가상 키보드

그 외 다른 인터페이스

→ 음성 명령, 제스처

3. System Call

OS에서 제공하는 서비스에 대한 프로그래밍 인터페이스 
{: .prompt-tip }

  • 일반적으로 고급 언어(C 또는 C++)로 작성
  • 대부분 프로그램에서 직접 시스템 호출을 사용하기보다는 높은 수준의 API(애플리케이션 프로그래밍 인터페이스) 를 통해 액세스
  • 가장 일반적인 세 가지 API 
    1. Windows용 Win32 API
    2. POSIX 기반 시스템용 POSIX API(거의 모든 버전의 UNIX, Linux 및 Mac OS X 포함)
    3. Java 가상 머신(JVM)을 위한 Java API

     

  • 이 텍스트 전체에 사용된 시스템 호출 이름은 일반적

한 파일의 내용을 다른 파일로 복사하기 위한 시스템 호출 시퀀스

{: .prompt-info }

구현

  • 일반적으로 각 시스템 호출에는 숫자가 연관

    → 시스템 호출 인터페이스는 이 번호에 따라 색인된 테이블을 유지
  • 시스템 호출 인터페이스는 OS 커널에서 의도한 시스템 호출을 호출하고 시스템 호출의 상태와 모든 반환 값을 반환
  • 호출자는 시스템 호출이 어떻게 구현되는지 알 필요가 없음

    → API를 준수하고 결과 호출로 OS가 무엇을 수행할지 이해하기만 하면 됨

    → 프로그래머가 알 수 없는 OS 인터페이스의 대부분의 세부 사항은 API에 의해 숨겨짐 런타임 지원 라이브러리(컴파일러에 포함된 라이브러리에 내장된 함수 집합)에 의해 관리됨
  • API + 시스템 호출 + OS 관계

Parameter Passing

  • 원하는 시스템 호출의 신원보다 더 많은 정보가 필요한 경우가 많음

    → 정확한 유형과 정보의 양은 OS와 호출에 따라 다릅니다. 
  • OS에 파라미터(매개변수)를 전달하는 데 사용되는 일반적인 3가지 방법
    1. 가장 간단한 방법: 레지스터에 매개변수 전달

      → 경우에 따라 레지스터보다 더 많은 파라미터가 있을 수 있음 
    2. 메모리의 블록 또는 테이블에 저장된 파라미터와 레지스터에 파라미터로 전달된 블록의 주소
      • 리눅스와 솔라리스가 취하는 접근 방식
      • 파라미터는 프로그램에 의해 스택에 배치되거나 푸시되고 운영 체제에 의해 스택에서 튀어나옵니다.

       

    3. 블록 및 스택 방식은 전달되는 파라미터의 수나 길이를 제한하지 않음

  • 파일 속성 가져오기 및 설정

3. 장치 관리

  • 장치 요청, 장치 해제
  • 읽기, 쓰기, 위치 변경 
  • 장치 속성 가져오기, 장치 속성 설정
  • 논리적으로 장치 연결 또는 분리

4. 정보 유지/관리

  • 시간 또는 날짜 가져오기, 시간 또는 날짜 설정
  • 시스템 데이터 가져오기, 시스템 데이터 설정
  • 프로세스, 파일 또는 장치 속성 가져오기 및 설정하기

5. 통신

  • 통신 연결 생성, 삭제 
  • 호스트 이름 또는 프로세스 이름으로 메시지 전달 모델인 경우 클라이언트에서 서버로 메시지 보내기, 받기 
  • 공유 메모리 모델 생성 및 메모리 영역에 대한 액세스 권한 부여
  • 상태 정보 전송
  • 원격 장치 연결 및 분리

6. 보호

  • 자원 접근
  • 권한
  • 사용자 액세스 허용 및 거부

4. 시스템 서비스

시스템 프로그램 또는 시스템 유틸리티는 프로그램 개발 및 실행을 위한 편리한 환경을 제공합니다.
{: .prompt-tip}

  • 다음의 환경들을 제공함
    1. 파일관리
    2. 상태정보
    3. 파일변경
    4. 프로그래밍 언어 지원
    5. 프로그램 로딩과 실행
    6. 통신
    7. 백그라운드리소스
  • 운영 체제에 대한 대부분의 사용자 관점은 실제 시스템 호출이 아닌 시스템 프로그램에 의해 정의
  • 프로그램 개발 및 실행을 위한 편리한 환경 제공
  • 일부는 단순히 시스템 호출에 대한 사용자 인터페이스이지만 다른 일부는 훨씬 더 복잡
  • 종류
    1. 파일 관리: 파일 및 디렉터리 생성, 삭제, 복사, 이름 변경, 인쇄, 덤프, 목록 작성 및 일반적인 조작이 가능
    2. 상태 정보
      • 일부는 날짜, 시간, 사용 가능한 메모리 양, 디스크 공간, 사용자 수 등의 정보를 시스템에 요청
      • 기타 세부 성능, 로깅 및 디버깅 정보 제공
      • 일반적으로 이러한 프로그램은 터미널 또는 기타 출력 장치에 출력을 포맷하고 인쇄
      • 일부 시스템은 구성 정보를 저장하고 검색하는 데 사용되는 레지스트리를 구현합
    3. 파일 수정
      • 파일 생성 및 수정을 위한 텍스트 편집기
      • 파일 내용을 검색하거나 텍스트 변형을 수행하는 특수 명령어
    4. 프로그래밍 언어 지원
      • 때때로 제공되는 컴파일러, 어셈블러, 디버거 및 인터프리터
    5. 프로그램 로딩 및 실행
      • 앱솔루트 로더, 재배치 가능한 로더, 링크 편집기, 오버레이 로더, 상위 수준 및 기계어를 위한 디버깅 시스템
    6. 커뮤니케이션
      • 사용자가 서로의 화면에 메시지를 보내고, 웹 페이지를 탐색하고, 전자 메일 메시지를 보내고, 원격으로 로그인하고, 한 시스템에서 다른 시스템으로 파일을 전송할 수 있도록 허용합니다.
    7. 백그라운드 서비스
      • 부팅 시 실행
      • 일부는 시스템 시작 시 실행 후 종료
        → 일부는 시스템 부팅부터 종료 시까지 실행됨
      • 디스크 검사, 프로세스 스케줄링, 오류 로깅, 인쇄와 같은 기능 제공
        → 커널 컨텍스트가 아닌 사용자 컨텍스트에서 실행됨
      • 서비스, 서브시스템, 데몬으로 알려져 있음
    8. 시스템 프로그램과 함께 애플리케이션 프로그램이 제공됨
      → 일반적인 문제를 해결하거나 일반적인 작업을 수행함
      • 사용자가 실행
      • 일반적으로 OS의 일부로 간주되지 않음
      • 명령줄, 마우스 클릭, 손가락 찌르기 등으로 실행됨

5. Linker 및 Loaders

  • 모든 물리적 메모리 위치에 로드할 수 있도록 설계된 오브젝트 파일로 컴파일된 소스 코드

    → 재배치 가능한 오브젝트 파일
  • Linker(링커) 는 단일 바이너리 실행 파일로 결합(실행 가능한 한 단위로 결합)

    → 라이브러리도 가져옴
  • 프로그램은 바이너리 실행 파일로 보조 저장소에 존재
  • 실행을 위해 Loaders(로더) 에 의해 메모리로 가져와야 함

    → Relocation(재배치) 는 프로그램 부분에 최종 주소를 할당하고 해당 주소에 맞게 프로그램 내 코드와 데이터를 조정
  • 최신 범용 시스템은 라이브러리를 실행 파일에 링크하지 않는 대신 동적으로 링크된 라이브러리(Windows의 경우 DLLs)는 필요에 따라 로드되며, 동일한 라이브러리의 동일한 버전을 사용하는 모든 라이브러리가 공유함(한 번 로드됨)
  • 객체, 실행 파일에는 표준 형식이 있으므로 운영 체제가 해당 파일을 로드하고 시작하는 방법을 알고 있음

6. 애플리케이션이 운영 체제별로 다른 이유

  • 한 시스템에서 컴파일된 앱은 일반적으로 다른 운영 체제에서 실행할 수 없음
  • 각 운영 체제는 고유한 시스템 호출(고유한 파일 형식 등)을 제공
  • 앱은 다중 운영 체제일 수 있습니다.
    • 여러 운영 체제에서 사용할 수 있는 파이썬, 루비 같은 인터프리터 언어로 작성됨
    • 실행 중인 앱이 포함된 VM을 포함하는 언어로 작성된 앱(예: Java)
    • 표준 언어(예: C) 사용, 각 운영 체제에서 개별적으로 컴파일하여 각 운영 체제에서 실행 가능
  • 애플리케이션 바이너리 인터페이스(ABI) 는 API에 해당하는 아키텍처입니다.
    • 바이너리 코드의 다양한 구성 요소가 주어진 운영 체제에 대해 어떻게 인터페이스할 수 있는지 정의
    • 바이너리 코드의 다양한 구성요소를 정의
    • 바이너리 코드의 다양한 구성요소가 지정된 아키텍처, CPU 등에서 특정 운영체제에 대해 어떻게 인터페이스 하는지 정의
  • 각 운영체제에는 헤더, 명령어, 변수의 레이아웃을 지시하는 애플리케이션용 바이너리 형식
  • CPU는 다양한 명령어 세트를 가지고 있으며, 적절한 명령어가 포함되어 있어야 애플리케이션이 올바르게 실행될 수 있음
  • OS에 대한 System calls은 다양함
    • 특정 피연산자 및 피연산자 순서
    • 애플리케이션이 시스템 호출을 호출하는 방법
    • 시스템 호출 번호와 숫자, 그 의미 및 반환 결과
  • ARM vs X86 - 명령어 집합 아키텍처

1. OS의 역할

운영체제(Operating System = os): 컴퓨터 사용자와 하드웨어 사시에서 중개자 역할을 하는 프로그램

  • 목표
    1. 사용자 프로그램을 실행하고 사용자 문제를 쉽게 해결
    2. 컴퓨터 시스템을 편리하게 사용
    3. 컴퓨터 하드웨어를 효율적으로 사용

컴퓨터 시스템의 네 가지 구성 요소

  1. Hardware
  2. OS
    1. 시스템 프로그램: OS와 함께 제공되지만 OS의 일부가 아닌 커널
    2. Middelware: DB, 멀티미디어, 그래픽과 같은 응용 프로그램 개발자에게 추가 서비스를 제공하는 software framework
    3. 커널: 일반적으로 항상 실행되는 프로그램
  3. Application programs: 사용자의 컴퓨팅 문제를 해결하기 위해 시스템 리소스가 사용되는 방식을 정의
  4. 사용자

2. 컴퓨터 시스탬 구성

  • 하나 이상의 CPU와 장치 컨트롤러는 공유 메모리에 접근이 가능한 공통 버스를 통해 연결되며, 병렬로 실행되어 메모리 사이클을 놓고 경쟁한다.
  • Device Controller(HW)[^1]는 CPU와 특정 장치 사이의 통신 중개를 담당하는 장치
  • OS는 각 장치 컨트롤러 유형마다 Device Driver(Program) 가 있고 해당 장치를 사용할 수 있는 인터페이스를 제공
  • CPU는 주 메모리 ↔︎ 로컬 버퍼로 데이터를 이동함
  • I/O 장치 → 컨트롤러의 로컬 버퍼로 데이터 이동 → 장치 컨트롤러는 interrupt를 발생

  1. 인터럽트의 일반적인 기능

    1. 인터럽트는 일반적으로 모든 서비스 루틴의 주소가 포함된 인터럽트 백터를 통해 인터럽트 서비스 루틴으로 제어권을 전송함
    2. 인터럽트 아키텍터는 인터럽트된 명령어의 주소를 저장해야 함
    3. 트랩 또는 예외는 오류 또는 사용자 요청으로 인해 발생하는 소프트웨어 인터럽트
    4. OS는 인터럽트 중심

  2. 인터럽트 처리

    • 운영 체제는 레지스터와 프로그램 카운터를 저장하여 CPU의 상태를 보존
    • 어떤 유형의 인터럽트가 발생했는지 결정하고 개별 코드 세그먼크는 이에 대해 수행할 작업을 결정

  3. 인터럽트 주도 I/O 사이클

  4. 스토리지 구조

    1. 주기억 장치: CPU가 직접 엑세스 할 수 있는 유일한 대용량 저장메체
      • 휘발성
      • 동적 랜덤 액세스 메모리(DRAM) 형태의 랜덤 액세스 메모리(RAM)**
    2. 보조기억 장치: 대용향 비휘발성(nonvolatile)저장소를 제공하는 메인 메모리의 확장
      • 비휘발성
      • 다양한 기술로 용량 및 성능이 증가, 가격 하락에 따라 대중화

  5. 스토리지 계층

    • 계층 구조로 구성된 스토리지 시스템 + 속도, 비용, 변동성
    • 매인 메모리: CPU가 직접 액세스할 수 있는 유일한 대용량 저장 매체
      1. Random access
      2. 휘발성: 전원이 꺼지면 내용을 잃음
      3. 일반적으로 Dynamic Random Access Memory 형태의 random access 메모리를 사용함
    • 보조 매모리: 대용량 비휘발성 저장소를 제공하는 메인 메모리의 확장
    • 캐싱: (반복적인 작업 등을) 더 빠른 스토리지 시스템 구성
      → 메인 메모리를 보조 스토리지의 캐시로 볼 수 있음
    • 각 디바이스 컨트롤러를 위한 디바이스 드라이버로 I/O 관리 가능
      → 컨트롤러와 커널 간의 일관된 인터페이스 제공

3. 컴퓨터 시스템 아키택쳐

  • 사용된 범용 프로세서의 수로 구분 + 싱글/멀티 프로세서
  • 멀티 프로세서 시스템: 멀티프로세서의 장점 + 처리량 증가, 규모의 경제(경제성), 신뢰성(안정성)[^2] 향상
  • 두가지 유형
    1. 비대칭 멀티프로세싱: 각 프로세서에 지정된 작업이 할당
    2. 대칭 멀티프로세싱: 각 프로세서가 모든 작업을 수행

듀얼코어 디자인

  • 멀티 칩 및 멀티코어
  • 모든 칩을 포함하는 시스템 + 여러 개의 개별 시스템을 포함하는 섀시

4. OS 작업

  • OS는 프로그램이 실행되는 환경을 제공
  • Bootstrap program: 시스템을 토기화 하고 커널을 로드하는 간단한 코드로 컴퓨터 하드웨어에 펌웨어로 저장됨
  • 커널 로드
  • 시스템 데몬(커널 외부에서 제공되는 서비스) 시작

커널 인터럽트 구동 (하드웨어 및 소프트웨어)

  • 장치 중 하나에 의한 하드웨어 인터럽트
  • 소프트웨어 인터럽트 (예외 or 트랩)
    1. 소프트 웨어 오류 (e.g., 0으로 나누기)
    2. OS 서비스 요청: System Call
    3. 기타 프로세스 문제: 무한 선순한 루프, 프로세스 간 또는 운영 체제 수정 등

멀티프로그래밍(배치 시스탬)

  • 프로세스(process): 실행 중인 프로그램, 시스템 내의 작업 단위, 프로그램을 수동적인 엔티티이고 프로세스는 능동적인 엔티티
  • 한 명의 사용자가 CPU와 I/O 장치를 항시 점유할 수 없음
  • 멀티프로그래밍은 작업(코드 및 데이터)을 구성하므로 CPU는 항상 실행할 작업이 하나씩 있음
  • 시스탬 내 전체 작업의 하위 집합이 메모리에 보관됨
  • 작업 스케줄링을 통해 하나의 작업 선택 및 실행
  • 작업이 대기해야 하는 경우(I/O) OS가 다른 작업으로 전환함

멀티테스킹(시간 공유)

  • 배치 시스탬의 논리적 확장: CPU가 작업을 자주 전환하여 사용자가 각 작업이 실행되는 동안 상호 작용하여 대화형(interactive) 컴퓨팅을 생성할 수 있음
  • *응답시간(response time)*이 1초 미만이어야 함
  • 각 사용자는 메모리 → 프로세스에서 실행 중인 프로그램이 하나 이상 있음
  • 동시에 여러 개의 작업을 실행할 준비가 된 경우 → CPU 스케줄링
  • 프로세스가 메모리에 맞지 않는 경우 스왑을 통해 프로세스를 안팎으로 이동하여 실행
  • 가상 메모리를 통해 프로세스 실행 가능

멀티프로그래밍 시스템을 위한 메모리 레이아웃


듀얼 모드 작동

OS는 자신과 다른 시스템 구성 요소를 보호할 수 있음
→ User 모드와 Kernel 모드
{: .prompt-tip }

  • 하드웨어에서 제공하는 모드 비트
    • 시스템이 사용자 코드 또는 커널 코드를 실행 중인지 구분할 수 있는 기능 제공 코드
    • 사용자가 실행 중일때는 모드비트가 사용자이며, 커널 코드가 실행 중일때는 모드비트는 커널

      → System call 은 모드를 커널로 변경, 호출에서 반환하면 사용자로 재설정함

    • 권한이 있는것으로 지정된 일부 명령어는 커널 모드서만 실행 가능.
  1. 타이머: 무한 루프를 방지하는 타이머(또는 프로세스 호깅 리소스)
    • 타이머는 일정 시간이 지나면 컴퓨터를 중단하도록 설정
    • 물리젃 시계에 의해 감소하는 카운터 유지
    • OS가 카운터 설정(권한이 있는 명령어)
    • 카운터 0이 인터럽트를 생성하는 경우 할당된 시간을 초과하는 프로그램을 다시 제어하거나 종료하기 위해 프로세스를 예약하기 전에 설정

5. 자원 관리

  • 프로세스는 작업을 수행하기 위해 리소스[^4]가 필요
  • 프로세스를 종료하려면 재사용 가능한 리소스를 회수해야 함
  • 단일 스레드 프로세스에는 실행할 다음 명령어의 위치를 지정하는 하나의 프로그램 카운터가 존재

    → 프로세스가 완료될 떄까지 한번에 순차적으로 명령어를 실행
  • 멀티 스레드 프로세스에는 스레드당 하나의 프로그램 카운터가 존재
  • 일반적으로 시스템에는 많은 프로세스, 일부 사용자, 일부 운영 체제가 하나 이상의 CPU에서 동시에 실행됨

    → 프로세스/스레드 간 CPU를 다중화하여 동시성 제공

프로세스 관리 활동

  • 운영 체제는 프로세스 관리와 관련하여 다음과 같은 활동을 담당
    1. 사용자 프로세스 및 시스템 프로세스 생성 및 삭제
    2. CPU의 프로세스 및 스레드 스케줄링
      3 프로세스 일시 중단 및 재개
    3. 프로세스 동기화를 위한 메커니즘 제공
    4. 프로세스 통신을 위한 메커니즘 제공

메모리 관리

  • 프로그램을 실행하려면 명령어의 전부(또는 일부)가 메모리에 있어야 함
  • 프로그램에 필요한 데이터의 전부(또는 일부)가 메모리에 있어야 함
  • 메모리 관리는 메모리에 무엇이 언제 있는지 결정

    → CPU 사용률 및 사용자에 대한 컴퓨터 응답 최적화
  • 메모리 관리 활동
    • 현재 사용 중인 메모리 부분과 해당 부분을 사용하는 프로세스를 추적합니다.
    • 필요에 따라 메모리 공간 할당 및 할당 해제
    • 메모리 안팎으로 이동할 프로세스(또는 이부 프로세스) 및 데이터 결정 메모리 안팎으로 이동

파일 시스템 관리

  • OS는 정보 스토리지에 대한 일관되고 논리적인 뷰 제공
  • 논리적 저장 단위인 파일로 물리적 속성을 추상화
  • 각 매체는 장치(디스크 드라이브, 테이프 드라이브)에 의해 제어됨

    → 엑세스 속도, 용량, 데이터 전송률, 엑세스 방법(순차 혹은 랜덤) 등 다양한 속성 포함
  • 일반적으로 디렉터리로 구성된 파일
  • 대부분의 시스템에서 누가 무엇에 엑세스할 수 있는지 결정하기 위한 엑세스 제어
  • OS 활동에는 다음이 포함됨
    1. 파일 생성 및 삭제
    2. 파일을 정리하기 위한 디렉터리 기본 요소 지원
    3. 파일 및 디렉터리 조작을 위한 기본 요소 지원
    4. 대용량 스토리지에 파일 매핑
    5. 안정적인(비휘발성) 스토리지 미디어에 파일 백업하기

대용량 스토리지(Mass Storage) 관리

  • 일반적으로 디스크는 주 메모리에 맞지 않는 데이터나 장기간 보관해야 하는 데이터를 저장하는데 사용됨
  • 적절한 관리가 가장 중요
  • 컴퓨터 작동의 전체 속도는 디스크 하위 시스템과 그 알고리즘에 달려 있음
  • OS 활동
    1. 마운팅 및 마운트 해제
    2. 여유 공간 관리
    3. 스토리지 할당
    4. 디스크 스케쥴링
    5. 파티셔닝
    6. 보호

캐싱

  • 컴퓨터의 여러 수준(하드웨어, 운영체제, 소프트웨어)에서 수행되는 중요한 원칙
  • 사용 중인 정보를 느린 스토리지에서 빠른 스토리지로 일시적으로 복사하는 것
  • 정보가 있는지 확인하기 위해 더 빠른 저장소(캐시)를 먼저 확인해서
    → 있는 경우, 캐시에서 직접 정보 사용(빠름)

    → 없는 경우 데이터를 캐시로 복사하여 캐시에서 사용
  • 캐시되는 스토리지보다 작은 캐시
    → 캐시 관리가 중요한 설계 문제

    → 캐시 크기 및 교체 정책

다양한 유형의 스토리지 특징

→ 스토리지 계층 구조의 레벨 간 이동은 명시적이거나 암시적일 수 있음

디스크

→ 레지스터로 데이터 “A”의 마이그레이션

  • 멀티테스킹 환경에서는 스토리지 계층 구조에 저장된 위치에 관계없이 가장 최근 값을 사용하도록 주의
  • 멀티프로세서 환경은 모든 CPU가 캐시에 가장 최근 값을 갖도록 하드웨어에서 캐시 일관성을 제공
  • 분산 환경의 상황은 훨씬 더 복잡함

    → 데이터의 복사본이 여러 개 존재할 수 있음

I/O 하위시스템

  • OS의 한 가지 목적은 하드웨어 장치의 특성을 사용자에게 숨기는 것입니다.
  • I/O 하위 시스템은 여러 구성 요소로 구성됩니다.
    1. 버퍼링, 캐싱, 스풀링이 포함된 메모리 관리 구성 요소
    2. 일반 장치+드라이버 인터페이스
    3. 특정 하드웨어 장치용 드라이버

6. 보안과 보호

  • 보안: 내부 및 외부 공격[^5]에 대한 시스템 방어
  • 보호: OS에서 정의한 리소스에 대한 프로세스 또는 사용자의 액세스를 제어하는 모든 메커니즘
  • 시스템은 일반적으로 먼저 사용자를 구분하여 누가 무엇을 할 수 있는지 결정합니다
    • 사용자 ID(사용자 ID, 보안 ID)에는 사용자당 하나씩 이름과 관련 번호가 포함되고, 그런 다음 사용자 ID를 해당 사용자의 모든 파일, 프로세스와 연결하여 액세스 제어를 결정
    • 그룹 식별자(그룹 ID)를 통해 사용자 집합을 정의하고 제어를 관리한 다음 각 프로세스, 파일과도 연결할 수 있음
    • 권한 에스컬레이션을 통해 사용자가 더 많은 권한을 가진 유효한 ID로 변경할 수 있음

7. 가상화

OS가 다른 OS 내에서 애플리케이션을 실행할 수 있음

→ 방대하고 성장하는 산업
{: .prompt-tip }

  • 소스 CPU 유형이 대상 유형과 다른 경우 에뮬레이션 사용(예: 애플 실리콘 로제타)

    → 일반적으로 가장 느린 방법

    → 컴퓨터 언어가 네이티브 코드로 컴파일되지 않은 경우: 인터프리터링
  • CPU용으로 기본 컴파일된 OS, 실행 중인 게스트 OS도 기본 컴파일됨

    → 기본 WinXP 호스트 OS에서 각각 애플리케이션을 실행하는 WinXP 게스트를 실행하는 VMware를 예로, VMM(가상 머신 관리자)이 가상화 서비스 제공
  • 사용 사례에는 탐색 또는 호환성을 위해 여러 OS를 실행하는 랩톱 및 데스크톱이 포함
    1. Mac OS X 호스트로 실행되는 Apple 노트북, 게스트로 실행되는 Windows
    2. 여러 시스템을 보유하지 않고 여러 OS용 앱 개발
    3. 여러 시스템을 사용하지 않고 애플리케이션을 테스트하는 품질 보증
    4. 데이터 센터 내 컴퓨터 환경 실행 및 관리
  • 컴퓨팅 환경 + 가상화

8. 커널 자료구조

표준 프로그래밍 데이터 구조와 유사한 많은 것들

  1. 싱글 링크 목록

  2. 이중 링크 목록

  3. 순환 연결 목록

스택과 큐

동등하게 정렬된 데이터 구조
{: .prompt-tip }

  1. 스택

    • 후입선출 방식(LIFO)
    • 스택에서 항목 삽입 및 제거를 위한 푸시 앤 팝 스택에서 항목 제거
    • 매개변수, 지역 변수, 반환 주소는 반환 주소가 스택에서 푸시 및 팝

  2. 큐

    • 선입선출(FIFO) 원칙

      → ex) 줄을 서서 기다리는 작업, 프린터로 전송된 작업

이진 검색 트리

  • 검색 성능은 O(n)
  • 균형 이진 검색 트리는 O(lg n)입니다.
  • 리눅스는 CPU 스케줄링 알고리즘의 일부로 균형 잡힌 이진 검색 트리를 사용

9. 컴퓨팅 환경

전통적인 방식

  • 독립형 범용 머신
  • 대부분의 시스템이 다른 시스템(예: 인터넷)과 상호 연결되므로 구분이 모호함
  • 포털은 내부 시스템에 대한 웹 액세스를 제공합니다.
  • 네트워크 컴퓨터(씬 클라이언트)**는 웹 터미널과 유사함
  • 모바일 컴퓨터가 무선 네트워크를 통해 상호 연결됨
  • 네트워킹의 보편화: 가정용 시스템에서도 방화벽 사용 방화벽을 사용하여 가정용 컴퓨터를 인터넷 공격으로부터 보호합니다.

모바일

  • 휴대용 스마트폰, 태블릿 등 + 기능적 차이점은 무엇인가요?
  • 일반 노트북과의 기능적 차이
    1. 추가 기능: 더 많은 OS 기능(GPS, 자이로스코프)
    2. 증강 현실과[^6] 같은 새로운 유형의 앱 사용 가능
    3. 연결을 위해 IEEE 802.11 무선 또는 셀룰러 데이터 네트워크 사용
  • 주요 운영체제: Apple iOS 및 Google Android

클라이언트 서버

  • 스마트 PC로 대체된 덤 단말기
  • 이제 많은 시스템이 서버가 되어 클라이언트가 생성한 요청에 응답
  • 컴퓨팅+서버 시스템은 클라이언트가 서비스(예: 데이터베이스)를 요청할 수 있는 인터페이스를 제공합니다.
  • 파일 서버 시스템은 클라이언트가 파일을 저장하고 검색할 수 있는 인터페이스를 제공합니다.

P2P

  • 분산 시스템의 또 다른 모델
  • 클라이언트와 서버를 구분하지 않음
    → 대신 모든 노드가 피어로 간주

    → 각 노드는 클라이언트, 서버 또는 둘 다의 역할 수행 가능

    → 노드는 P2P 네트워크에 가입해야 함

    → 네트워크의 중앙 조회 서비스에 서비스를 등록하거나, 서비스 요청을 브로드캐스트하고 검색 프로토콜을 통해 서비스 요청에 응답

    → ex) 냅스터(Napster), 누텔라(Gnutella), 다음과 같은 VoIP(Voice over IP) Skype

클라우드 컴퓨팅

  • 컴퓨팅, 스토리지, 심지어 서비스형 애플리케이션까지 네트워크를 통해 제공
  • 가상화를 기능의 기반으로 사용하기 때문에 가상화의 논리적 확장

    → 아마존 EC2는 수천 대의 서버, 수백만 대의 가상 머신, 페타바이트 규모의 스토리지를 인터넷에서 사용할 수 있으며, 사용량에 따라 요금이 부과
  • 다양한 유형
    1. 퍼블릭 클라우드: 비용을 지불하면 누구나 인터넷을 통해 사용 가능
    2. 프라이빗 클라우드: 회사에서 회사 자체용으로 운영하는 클라우드
    3. 하이브리드 클라우드: 퍼블릭 클라우드와 프라이빗 클라우드 구성 요소가 모두 포함됨
    4. 서비스형 소프트웨어(SaaS): 인터넷을 통해 사용할 수 있는 하나 이상의 애플리케이션(예: 워드 프로세서)
    5. 서비스형 플랫폼(PaaS): 인터넷을 통해 애플리케이션을 사용할 수 있는 소프트웨어 스택(예: 데이터베이스 서버)
    6. 서비스형 인프라(IaaS): 인터넷을 통해 사용 가능한 서버 또는 스토리지 인터넷(즉, 백업용으로 사용할 수 있는 스토리지)
  • 기존 OS, VMM, 클라우드 관리 툴로 구성된 클라우드 컴퓨팅 환경
    → 인터넷 연결에는 방화벽과 같은 보안이 필요합니다.
    → 로드 밸런서는 여러 애플리케이션에 트래픽을 분산시킵니다.
  • 클라우드 서비스 모델

실시간 임베디드 시스템

  • 가장 널리 사용되는 컴퓨터 형태
  • 다양한 특수 목적, 한정된 목적의 OS, 실시간 OS

    → 용도 확장 중
  • 기타 여러 특수 컴퓨팅 환경에서도 사용 가능

    → 일부에는 OS가 있고, 일부는 OS 없이 작업을 수행
  • 실시간 OS에는 잘 정의된 고정 시간 제약이 존재

    → 제약 조건 내에서 처리를 수행해야 하며, 충족되는 경우에만 올바른 작업 수행
  • 사물인터넷(IoT: Internet of Things) 기술: 사람, 사물, 데이터 등 모든것이 유무선 인터넷으로 연결되어, 정보가 생성(센서)·수딥(부품·디바이스)·공유(클라우드)·활용(빅데이터·응용SW)되는 기술과 서비스를 통칭하는 개념으로 주로 자동차분야, 농업분야(과수원, 재해, 화훼 등), 헬스케어분야, 사회안전 분야 등에 활용
    1. 센싱기술: 온도, 습도, 열, 가스, 위치, 속도, 및 도조 등과 같은 물리량을 다양한 방법을 이용해 측정하는 기술

      → 인터넷 공유를 위해 신호처리 및 알고리즘 수행이 가능한 모듈이 내장된 스마트 센서 기술
    2. 인터페이스 기술: 사물인터넷의 주요 구성요소(인간·사물·서비스)를 통해 특정 기능을 수행하는 응용서비스와 연동하는 기술
    3. 네트워킹 기술: 다양한 디바이스들의 물리적인 연결을 수행 하는 유무선 네트워킹 기술
